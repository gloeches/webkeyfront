Git Recovery Guide

1. How to Revert Local Uncommitted Changes

This is for when you have made changes to files in your project, have not committed them, and want to discard them completely.

git reset --hard HEAD

What it does: This command resets your entire project to match the last commit. It discards all changes in your working directory and staging area.

Warning: This action is destructive and cannot be undone. All of your uncommitted local changes will be permanently lost.

Alternative:

git checkout -- .

This is a slightly less powerful command that only discards changes in the working directory, but not the staging area.

2. How to Revert to a Specific Older Commit

This is for when you want to go back in time to a specific version of your project that you have already committed.

Step 1: Find the Commit Hash

First, you need to find the unique identifier (hash) of the commit you want to go back to. You can see the project history with:

git log --oneline

This will show you a list of commits, each with a short hash (e.g., 3d7ee29).

Step 2: Reset to the Commit

Once you have the hash, you can reset your project to that state.

git reset --hard <commit_hash>

For example:

git reset --hard 3d7ee29
Warning: This rewrites your project's history. All commits made *after* the one you reset to will be removed from your local branch. This can be very problematic if you have already shared those commits on GitHub.

3. How to Revert to Match the GitHub Repository

This is for when your local project has changes and/or commits that you want to completely discard, making your local branch an exact copy of the remote repository on GitHub.

Step 1: Fetch the Latest State from GitHub

First, download the latest information from the remote repository (named origin by default).

git fetch origin

Step 2: Reset Your Local Branch to Match the Remote

Now, force your local branch to be identical to the one you just fetched. You will need to specify the branch name, which is commonly master or main.

git reset --hard origin/master

(If your main branch is called main, use origin/main instead)

Warning: This is highly destructive. It throws away all local changes and commits that you have not pushed to GitHub. Your local branch will become an exact replica of the remote branch.

4. How to Check Out a Specific Version from GitHub (Non-Destructive)

Use these commands when you want to safely look at or build an old version of your code without losing your current work.

Info: The following git checkout commands will put you in a state called a "detached HEAD". This means you are viewing a snapshot of your project's history, but you are not on a formal branch. It's safe for inspection, but if you want to save any new work, you must create a new branch.

4.1 From a Specific Commit

If you want to view the project as it existed at a specific commit:

  1. Fetch latest data: git fetch origin
  2. Checkout the commit: git checkout <commit_hash> (e.g., git checkout 3d7ee29)

Your files will now match that commit. To return to your main branch:

git checkout master

4.2 From a Specific Tag

Tags are often used to mark release versions (e.g., v1.2.0). This is the best way to get the code for a specific release.

  1. Fetch all tags from GitHub: 
    git fetch --all --tags
  2. List available tags (optional): 
    git tag
  3. Checkout the tag: 
    git checkout tags/<tag_name>
    (e.g., git checkout tags/v1.2.0)

Your files will now match that release tag. To return to your main branch:

git checkout master

What to do in a "Detached HEAD" state?

If you decide you want to make changes starting from this old version, you must create a new branch first:

git checkout -b <new_branch_name>

This creates a new branch with the old code, allowing you to commit your work safely without losing it.
